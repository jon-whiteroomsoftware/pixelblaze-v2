# Capability exploration spike (Phase 2, hardware connectivity)

Establishes **empirically how far the Pixelblaze WebSocket protocol goes** before any
UI is designed — the Phase-2 gate for the Hardware Connectivity feature (#108). Built on
the extended `PixelblazeConnection` comms layer (#106), which gained the binary-frame
protocol for this phase.

**This is a human-in-the-loop, out-of-band tool.** It needs a physical Pixelblaze on your
LAN and is *excluded from the pre-commit gate* (it touches the network and, optionally,
writes to flash).

## How to run

Point it at the device and note the firmware version (shown in the editor's settings):

```bash
PIXELBLAZE_IP=192.168.8.224 PIXELBLAZE_FW=3.67 npm run spike
```

It connects, exercises each capability, restores the device's original active program and
brightness, and writes **`report.md`**. Commit that report, then update `findings.md` with
any live-confirmed conclusions.

**Flash safety:** the spike never passes `save:true`. Control/brightness probes are
volatile (lost on reboot) by design, so repeated runs cost no flash wear.

## What it answers

- `listPrograms` — does the type-7 binary multi-frame decode into a real id/name list?
- `activeProgramId` — can we read the active program and switch it?
- `getControls` / `setControls` — round-trip, and does a value persist without `save`?
- `brightness` — set/confirm round-trip.
- **Pattern push** (`putSourceCode`) — the headline unknown. Source-only push is attempted
  and observed; the bytecode-compiler question that actually gates push is investigated in
  `findings.md`.

## Files

| file | role |
|---|---|
| `spike.ts` | the runner: connect → exercise each capability → write report |
| `report.md` | committed deliverable (auto-generated by a live run) |
| `findings.md` | hand-written interpretation, bytecode investigation, **gate recommendation** |

If a capability contradicts an assumption, or push proves viable/non-viable, file a
follow-up issue and link it from `findings.md`.
