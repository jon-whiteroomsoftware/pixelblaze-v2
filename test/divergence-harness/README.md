# Divergence harness (Phase 1, hardware connectivity)

Characterises the preview's built-ins **numerically against a real Pixelblaze**
and produces the committed `report.md`. Built on the framework-free
`PixelblazeConnection` comms layer (#106).

**This is a human-in-the-loop, out-of-band tool.** It needs a physical Pixelblaze
on your LAN and is *excluded from the pre-commit gate* (it touches the network).

## How to run

1. **Hand-load the probe.** Open the device's stock ElectroMage editor
   (`http://<device-ip>/`), paste in the contents of [`probe.js`](./probe.js),
   and save it. (Phase 1 deliberately avoids the undocumented binary push
   protocol — the pattern is loaded by hand, once.) Leave that pattern active.

2. **Run the harness**, pointing it at the device and noting the firmware version
   (shown in the editor's settings, e.g. `3.67`):

   ```bash
   PIXELBLAZE_IP=192.168.1.50 PIXELBLAZE_FW=3.67 npm run harness
   ```

3. It sweeps inputs via `setVars`, reads results via `getVars`, computes
   per-built-in divergence and behaviour answers, and writes **`report.md`**.
   Commit that report.

## What it answers

- Per-built-in divergence (max |Δ|) for the transcendentals.
- Whether the candidate integer hashes (`hash11`/`hash21` from Noise.js /
  Shader.js) are **bit-identical** preview↔hardware.
- Confirmed firmware behaviour: overflow **wrap vs saturate**, multiply
  **round vs truncate**, and `frac` / `mod` / `~` **negative-edge** handling.

## Files

| file | role |
|---|---|
| `probe.js` | Pixelblaze-dialect probe pattern — **hand-loaded onto the device** |
| `builtins-spec.ts` | what to probe, sweeps, and float64 / fixed-point references |
| `harness.ts` | the runner: connect → sweep → compare → write report |
| `report.md` | committed deliverable (generated by a live run) |

If the device contradicts a `fixedpoint.ts` / shim assumption, file a follow-up
issue and link it from `report.md`.
