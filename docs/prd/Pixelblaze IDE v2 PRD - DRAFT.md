Pixelblaze IDE PRD (Original Draft)

- Terminology
  - Pixelblaze
    - A hardware controller for LED lighting sold by [ElectroMage](https://electromage.com/)
    - The entire ecosystem of hardware, software etc. that ElectroMage provides
  - Pattern
    - A small code file to be run on a pixelblaze controller. Code is written in Pixelblaze's custom [JavaScript-derived language](https://electromage.com/docs/language-reference)
    - The LED light display that is the result of running the pattern code
- Why
  - The basic pattern editor provided by ElectroMage has a few major limitations:
    - 1 - Code editing is a bit primitive and missing some standard IDE nicities
    - 2 - No ability to work entirely offline building patterns without a connected controller
    - 3 - No code management or reuse. Patterns must have all code inline in the same file
- What
  - A React-based browser app. The server exists to get the files up and running on the user's browser only. There are no back-end APIs, storage etc. for this app.
- Features
  - Main UI Layout
    - Top Bar - Global status and navigation
    - Left pane - Pattern file picker that shows both library and user files in a list
    - Middle pane - Pattern code editor 
    - Right pane - Pattern preview grid (mimics real LED grid)
    - Vertical splitter control between middle and right panes
  - Offline mode
    - Build and run patterns within the IDE without needing a controller connected
  - Code Library
    - The IDE provides reusable code organized into multiple libraries to enable the user to writing pattern code quickly
    - Library code is focussed on performance given the strict hardware limitations of the controller
    - Libraries can reference code in other libraries, referencing them as static singletons, e.g. Animate.setCycle(3). User pattern code can refer to library functions the same way. Those references are all resolved when the pattern is transpiled together
    - Does not duplicate any of the built-in PIxelblaze functionality
    - Ideas for library code: scalar math helpers — polar coords, SDFs (circle, rect, polygon, segment), blend modes, value noise, palette interpolation, animation primitives
    - Library code is read-only. It can simply be stored as one library per file in a specific directory on the project's server.
    - The library files can be opened in the IDE editor window but they are read-only
  - User pattern storage
    - Users can make their own patterns that persist between IDE sessions
    - User patterns storage should utilize the user's own browser, e.g. via IndexDB or LocalStorage
  - Code Editing
    - Code parser
      - Parse pixelblaze's built-in language
      - Determine whether code is compiled correctly. Needs to correctly parse and resolve references to functions in code libraries
    - Code transpiler
      - Patterns that contain references to library code need to have those references resolved before the pattern can be run. The transpiler finds the referenced library functions (including those referenced transitively) and prepends them to the final code with one object for each top level 
      - Transpiler needs to produce a complete valid pattern that will be used in various destinations:
        - Uploaded into a hardware controller
        - In the pattern preview pane
        - Downloading a file to disk
    - Syntax highlighting
    - Code Auto-complete including standard library and IDE library functions
    - Signature hints: typing ( after a lib function shows parameter names with the active parameter highlighted; advances as commas are typed
    - Error detection
      - Background parsing of pattern code as user types
      - Good / Broken indicator
    - Periodic auto-save (as long as code is compiling)
  - Pattern preview
    - Show a grid of LEDs that can can run the pattern animation that's currently open in the editor (as long as it compiles).
    - User can also manually play or pause patterns
      - Show run-status with a pill toggle next to the pattern file name
      - Pill shows Stopped/Running/Error state and acts as play/pause toggle
      - Pattern dims when paused
      - Patterns load with a single preview frame rendered so the frozen state is visible
      - Running state is preserved across pattern switches: if running, new pattern plays immediately; if paused, stays paused with preview frame shown
      - App always starts paused when loaded
      - Error state always pauses
    - User can adjust the overall brightness of the display (from 0 to 1). Note that this setting also exists on the hardware controller
    - User can control number of LEDs in the grid (columns and rows), the spacing of the LEDs and the how much of a glow/blue effect is applied to the grid (or disable it entirely)
    - User can adjust the speed of the pattern playing making it slower or faster than actual, e.g. 0.1x 0.5x, 1x, 2x.
    - Pattern UI controls
      - A pattern can contain code that will render a UI affordance (e.g. a color picker) that will be rendered in the preview area to adjust pattern variables via sliders, toggles and color pickers.
  - Debugging
    - Near the pattern preview is a table of showing the names and current values of the built-in and user created variables that the pattern is using.
  - Hardware upload
    - Find and connect to pixelblaze controller on network
    - Use it's WebSocket API to read existing patterns, upload patterns, and interact with other APIs such as changing pattern variables
