# Surfaces are 2D viewport embeddings; the cylinder wraps a chosen map, not a stock map

**Status:** accepted — supersedes the cylinder stance in [ADR-0009](0009-maps-authoritative-true-aspect.md) and [ADR-0008](0008-map-functions-are-plain-javascript.md); generalizes [ADR-0005](0005-display-position-dual-sourced.md) from 1D to 2D. **Partly superseded by [ADR-0012](0012-three-embedding-mechanisms-shell-vs-volume.md): the Surface family is narrowed to developable wraps (Flat + Cylinder only). The "sphere, torus, polyhedron-net Surfaces" expectation below does NOT hold — a sphere needs a distortive projection and a cube net accepts only square-per-face grids, so those shapes live as 3D shell/volume maps instead, not as Surfaces.**

The cylinder was first shipped as a self-contained stock **map** with a fixed UV parameterization (`dim:2, displayDim:3`, `cols ≈ √(πN)`) — a shortcut taken while the preview was first coming up. ADR-0009 then parked "compose a custom map onto a cylinder" as deferred, and ADR-0008 carved the cylinder out as a no-source stock-map exception. Revisiting that properly: the cylinder is not a map at all. It is a **viewport embedding** that owns `pos` (where each dot is drawn in 3D) while the **map** keeps owning `sample` (the `[u,v]` the pattern reads) — the §`Sample / position` divergence (ADR-0005), made first-class for 2D. This is the exact 2D analogue of the 1D **shapes** (line/ring/pole): a shape wraps a 1D strip (empty `sample`), a **surface** wraps a 2D map's flat field.

## Decision

**A `Surface` is the 2D sibling of a `Shape`: a `pos`-only viewport embedding, composed with a source map.** The cylinder, sphere, torus, polyhedron nets are surfaces; the plain 2D preview is the trivial **Flat** (identity) surface — one family, not a special case.

- **Two orthogonal controls, not one union dropdown.** The retired single "Layout dropdown" splits into a **Map** control (owns `sample`) and an **embedding** control (owns `pos`). The embedding control is populated with **shapes** for a 1D pattern and **surfaces** for a 2D pattern. Controls show only when they carry a real choice: 1D → embedding only; 2D with a wrappable map → both (map left, surface right); 2D with an irregular map, or 3D → map only. Control count changes with the pattern's dimensionality, which is acceptable because dimensionality only changes when the user deliberately switches patterns in the left rail.
- **Surface geometry is fully map-derived; no slenderness knob.** The cylinder reads the map's **raw** integer `gridDims` (`cols×rows`), not its ADR-0009-normalized `sample` (which caps the longest axis at `[0,1]` and would flatten every map to circumference 1). `circumference:height = cols:rows`, so the unrolled tube *is* the map rectangle — a square map wraps to a tall slender tube (~π:1), a 2:1 map to a fatter, shorter one. "The map is authoritative" (ADR-0009) now governs surface shape too. The 1D **Pole** keeps its `cols` slenderness slider only because a 1D strip has no aspect to read.
- **Surfaces are offered only for maps with `gridDims`.** Wrapping the `cols` axis around the circumference requires a clean integer `cols`. The stock Square and regular-lattice custom maps qualify; an irregular cloud (a measured tree, a cross install) has no periodic axis, so it is offered **Flat** only — and a single-option embedding control is hidden, leaving the map control alone (consistent with "hide when not needed").
- **Default surface is Flat; the choice persists per-pattern** alongside `mapId` on `PatternRecord` (no `DB_VERSION` bump — schemaless record). Selecting a wrappable map never surprise-wraps.

## Considered options

- **Keep the cylinder as a self-contained stock map (status quo, ADR-0009).** Simplest; nothing new ships. Rejected: it can't honor a chosen map's aspect (a square and a 2:1 map look identical wrapped), it makes the cylinder a permanent special case in both the map catalogue and the ADR set, and it doesn't scale to the sphere/torus/polyhedron surfaces we expect to add.
- **One union dropdown with surfaces nested per map** (`Square`, `Square → Cylinder`, …). One control, no second knob. Rejected: the list grows multiplicatively (maps × surfaces) and conflates the two orthogonal choices (`sample` field vs `pos` embedding) the data model keeps separate.
- **Surface as a viewport-panel modifier** (beside light size / diffusion), Layout staying map-only. Rejected: the surface determines the pattern's whole spatial reading on screen — it belongs with map selection, not buried with cosmetic comfort prefs.
- **Offer surfaces for any 2D map, wrapping by bounding box.** Maximum freedom. Rejected: a non-periodic map's seam collides or gaps arbitrarily; gating on `gridDims` keeps the wrap meaningful and the offered set honest.
- **A free slenderness knob on the cylinder.** Rejected: it would let the user distort the map's true proportions, contradicting "the map is authoritative." 1D needs the knob only because it has no aspect; 2D does, so it reads it.

## Consequences

- **The stock cylinder map is retired** and re-expressed as the Cylinder surface plus the Square map. The `createCylinderMap` path (`src/engine/maps/cylinder.ts`) moves to a surface-embedding module beside the 1D shapes (`src/engine/shapes.ts`), reading a map's `gridDims` rather than generating its own grid. The shared π-cell math is now used by both the Pole shape and the Cylinder surface.
- **`layout.ts` gains an embedding axis.** `layoutOptions` currently returns one flat list; it must yield the two-control structure — the map list (filtered by sample-arity as today) plus the embedding list (shapes for 1D; surfaces gated on the active map's `gridDims` for 2D). `LayoutSelection` carries a `surfaceId` alongside `mapId`/`shapeId`.
- **ADR-0008's "drape cylinder no-source exception" dissolves.** With the cylinder no longer a stock map, there is no source-less stock-map case; the no-source carve-out is removed rather than maintained.
- **ADR-0005 generalizes.** Its "1D shapes are viewport embeddings" becomes "viewport embeddings span 1D shapes and 2D surfaces"; the cosmetic-vs-semantic rule (an embedding is cosmetic because `sample` is untouched) is unchanged — it now just applies one dimension up.
- **The glossary is updated** (`CONTEXT.md`): `Surface` added as the 2D embedding; `Shape` cross-referenced as its 1D sibling; `Layout` rewritten as the two-control collective; the single "Layout dropdown" union retired.
- **Sphere, torus, and polyhedron-net surfaces** are now straightforward catalogue additions to the embedding module — each is a `gridDims → pos` projection — rather than each being a bespoke stock map.
